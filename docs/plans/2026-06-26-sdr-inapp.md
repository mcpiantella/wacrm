# Blueprint — SDR in-app (qualificador conversacional por LLM)

**Objetivo:** construir o SDR (Sales Development Rep) conversacional **dentro do app** (wacrm fork), aposentando o workflow n8n `[ZG] SDR v5.0`. Foco declarado do dono: **escalabilidade multi-tenant**. Substitui o **D2/S9** do blueprint de fusão (`../superpowers/plans/2026-06-25-blueprint-merge-wacrm-zenith.md` no repo Disparador).

## Decisões travadas (input do dono)
- **D1 — Config por CAMPANHA** (broadcast), não por account. Cada campanha tem objetivo/prompt/critérios próprios.
- **D2 — Debounce 10–15s** (default 12s, configurável por campanha). Junta mensagens picadas numa só vez antes de responder.
- **D3 — Handoff MATA o SDR naquela conversa** (marca pra humano + notifica + para de responder ali).
- **D4 — Áudio: OpenAI `gpt-4o-mini-transcribe`** como primário (usa o crédito OpenAI do dono, 1 vendor), **atrás de interface pluggável** (Groq/Deepgram trocáveis).
- **D5 — LLM: `gpt-5-mini` → Claude fallback** (padrão já adotado no projeto).
- **D6 — Worker:** processo de longa duração como **serviço Easypanel separado, mesma imagem**, comando próprio. (Easypanel roda container → worker persistente é viável; serverless não seria.)

## Arquitetura-alvo
```
webhook (Cloud/Evolution) → processInboundMessage (já existe)
   └─ se conversa.sdr_status='active' e campanha tem SDR habilitado:
        enqueue/refresh job debounced (jobId = sdr:<conversationId>)
            ↓ (Redis / BullMQ, delay = debounce_seconds)
        Worker SDR (serviço separado)
          ├─ carrega: sdr_config da campanha + histórico + lead/contato + canal da conversa
          ├─ se última msg é áudio → transcreve (Transcriber pluggável)
          ├─ monta mensagens → LLM (gpt-5-mini → Claude fallback)
          ├─ parse: { reply, qualification, handoff }
          ├─ AÇÃO:
          │    reply   → createChannel(canal).sendText + persiste msg + bump conversa
          │    handoff → sdr_status='handoff', notifica, NÃO responde mais
          │    noop    → só loga
          └─ grava sdr_runs (observabilidade)
```

## SDR-1 — Port do provider de chat (`lib/ai`)

**Contexto cold-start.** O projeto antigo (Disparador, `mcpiantella/zenith_sender`) tem `src/lib/ai/providers.ts` com OpenAI principal (`gpt-5-mini`) e Claude como fallback. O wacrm fork ainda **não** tem `lib/ai`. Tanto o SDR (SDR-5) quanto a IA de campanha (S8 da fusão) precisam desse provider. Este passo porta **só** o chat provider puro.

**Tarefas.**
1. Criar `src/lib/ai/chat.ts`: função `chatComplete({ system, messages, model?, responseFormat? })` que tenta OpenAI (`OPENAI_MODEL` default `gpt-5-mini`) e, em erro/sem chave, cai pro Anthropic (`claude-sonnet-4-6`). SDKs `openai` e `@anthropic-ai/sdk` já estão nas deps do wacrm — confirmar com `node -e "require('./package.json')"`. **Referência opcional:** `mcpiantella/zenith_sender` → `src/lib/ai/providers.ts`. Se o repo antigo não estiver acessível ao agente, **escrever do zero** — a lógica é simples (1 chamada OpenAI, catch → 1 chamada Anthropic) e este brief é suficiente.
2. Suportar saída **estruturada** (JSON) — o SDR precisa de `{ reply, qualification, handoff }`. Usar JSON mode/response_format quando disponível; parsear defensivamente.
3. Ler chaves de env (`OPENAI_API_KEY`, `ANTHROPIC_API_KEY`); nunca hardcode. Atualizar `.env.example`.
4. Testes (vitest, mock dos SDKs): usa OpenAI quando há chave; cai pro Claude quando OpenAI lança; parse JSON; erro claro se ambos faltam.

**Exit:** `chatComplete` testado (fallback + JSON), tsc/eslint/build limpos, `.env.example` atualizado. Sem dependência de Supabase/fila.

## SDR-2 — Interface de transcrição + impl OpenAI

**Contexto cold-start.** Leads mandam áudio; o SDR precisa de texto. Decisão D4: OpenAI `gpt-4o-mini-transcribe` primário, atrás de interface trocável.

**Tarefas.**
1. `src/lib/ai/transcription/types.ts`: `interface Transcriber { transcribe(input: { url?: string; bytes?: ArrayBuffer; mimeType: string }): Promise<string> }`.
2. `src/lib/ai/transcription/openai-transcriber.ts`: implementa via OpenAI audio transcription (`gpt-4o-mini-transcribe`). Baixa o áudio (do `media_url` / Evolution media), envia, retorna texto. Idioma pt-BR como dica.
3. `factory.ts`: `getTranscriber()` por env (`TRANSCRIBER` default `openai`), pronta pra `groq`/`deepgram` depois (lançar "não implementado" claro).
4. Testes (mock fetch/SDK): retorna texto; erro claro em falha; factory resolve openai.

**Exit:** transcrição atrás de interface, OpenAI funcionando, testes, tsc/eslint/build limpos. Pura (sem fila/Supabase).

## SDR-3 — Infra de fila (Redis + BullMQ + debounce)

**Contexto cold-start.** O SDR é assíncrono: precisa juntar mensagens (debounce 10–15s) e processar fora do request. O app ainda não conecta no Redis. BullMQ sobre Redis é o mecanismo.

**Tarefas.**
1. Adicionar deps: `bullmq`, `ioredis`. `REDIS_URL` em env + `.env.example`.
2. `src/lib/queue/connection.ts`: conexão ioredis singleton (lazy, lê `REDIS_URL`). Tolerar ausência em build (não conectar em import-time).
3. `src/lib/queue/sdr-queue.ts`: define a fila `sdr`, tipo do payload `{ conversationId, accountId }`, e `enqueueSdr(conversationId, accountId, debounceSeconds)`:
   - **Debounce por conversa:** `jobId = sdr:<conversationId>`; antes de adicionar, remove o job atrasado existente daquela conversa e re-adiciona com `delay = debounceSeconds*1000`. Mensagens em rajada **resetam o timer** → uma execução só. `removeOnComplete`/`removeOnFail` ligados.
4. Testes da lógica de debounce/enqueue (mock do BullMQ): rajada gera 1 job; jobId estável; delay aplicado.

**Exit:** fila + enqueue debounced testados; `REDIS_URL` documentado; build não tenta conectar Redis em import-time; tsc/eslint limpos. **Não** consome ainda (sem worker).

**Nota de risco (M):** garantir que importar `sdr-queue` numa rota Next não abra conexão Redis no build/edge. Conexão deve ser lazy.

## SDR-4 — Schema: `sdr_configs`, `sdr_status`, `sdr_runs`  *(strongest)*

**Contexto cold-start.** Config por campanha (D1), estado por conversa, e log de execuções. Migrações wacrm vão até 028; criar `029_sdr.sql`. Padrão de RLS: `is_account_member(account_id)` (settings-class p/ config; member read p/ runs). Ver migração 028 como referência de estilo.

**Tarefas.**
1. `sdr_configs` (1:1 com broadcast/campanha): `id`, `account_id` (FK accounts, cascade), `broadcast_id` (FK broadcasts, cascade, **UNIQUE**), `enabled bool default false`, `system_prompt text`, `qualification_criteria jsonb default '[]'`, `model text`, `handoff_keywords text[] default '{}'`, `max_turns int default 20`, `debounce_seconds int default 12 CHECK (5..60)`, timestamps. RLS: select member; insert/update/delete `'admin'`. Trigger updated_at.
2. `conversations.sdr_status` enum-like text `CHECK in ('off','active','handoff')` default `'off'` + índice parcial em `('active')`.
2b. **Elo conversa→campanha (CRÍTICO):** `conversations.broadcast_id uuid REFERENCES broadcasts(id) ON DELETE SET NULL` + índice. É como o worker acha o `sdr_config` da campanha. **Primeiro checar se já existe mapeamento** (ex.: tabela `broadcast_recipients` — o wacrm tem `recompute_broadcast_counts`, indício de uma tabela de destinatários; `grep -rn "broadcast_recipients\|recipient" supabase/migrations`). Se houver mapeamento contato↔broadcast confiável, **resolver a campanha por ele** em vez de duplicar a coluna; caso contrário, usar `conversations.broadcast_id`.
2c. **Ativação:** definir onde `sdr_status` vira `'active'` e `broadcast_id` é setado — no momento em que o disparo cria/atualiza a conversa do destinatário (caminho de envio do broadcast). Como o disparo ainda está sendo migrado, deixar a regra explícita: SDR ativa quando (a) a campanha tem `sdr_config.enabled=true` e (b) a conversa está vinculada a essa campanha. Documentar o gatilho exato escolhido.
3. `sdr_runs` (log): `id`, `account_id`, `conversation_id` (FK, cascade), `broadcast_id` (FK set null), `inbound_message_ids uuid[]`, `transcript_in text`, `llm_output jsonb`, `action text CHECK in ('reply','handoff','noop','error')`, `reply_message_id uuid`, `tokens int`, `latency_ms int`, `error text`, `created_at`. RLS: select member; insert via service-role (worker) — sem policy de insert pro authenticated.
4. Aplicar via MCP no projeto `raqfuattunokpbozdhkc` **e** versionar `supabase/migrations/029_sdr.sql` (paridade — conferir com `pg_get_functiondef`/`information_schema` como nas migrações anteriores).
5. Rodar `get_advisors(security)` — **0 advisors novos**.

**Exit:** 3 objetos criados, RLS provada (membro lê config; não-membro bloqueado; admin escreve config), 0 advisors novos, migração versionada com paridade código↔banco.

## SDR-5 — Núcleo do SDR (orquestração pura)  *(strongest)*

**Contexto cold-start.** O "cérebro": dado um `conversationId`, decide o que fazer. **Puro e injetável** (recebe supabase admin client, transcriber, chatComplete como deps) → testável sem Redis/rede. Não envia nada nem mexe na fila; **retorna uma decisão**.

**Tarefas.**
1. `src/lib/sdr/core.ts`: `decideSdrAction(deps, { conversationId }): Promise<SdrDecision>`.
   - Carrega: conversa (+ `channel_id`, `sdr_status`, `broadcast_id`), campanha/broadcast vinculada **via o elo do SDR-4 (2b)**, `sdr_config` (enabled?), histórico de mensagens (ordenado), contato/lead. Se a conversa não tem campanha vinculada → `noop` (sem campanha, sem SDR — R6).
   - **Guardas (retornam noop/skip):** `sdr_status != 'active'`; config ausente/`enabled=false`; última mensagem é `fromMe`/agente (não responder a si mesmo); `max_turns` excedido.
   - Se a última mensagem do lead for áudio → `transcriber.transcribe(...)` e usar o texto.
   - Monta `system` (do `system_prompt` da campanha + critérios) + histórico → `chatComplete(..., responseFormat: json)`.
   - Parse defensivo: `{ reply: string, qualification: {...}, handoff: boolean }`. Handoff também dispara se `handoff_keywords` aparecerem ou critérios de qualificação baterem.
   - Retorna `SdrDecision`: `{ action: 'reply'|'handoff'|'noop', replyText?, qualification?, transcriptIn?, raw }`.
2. Testes pesados (é a lógica mais arriscada): guarda de self-message; handoff por keyword; handoff por sinal do LLM; reply normal; áudio→transcrição→reply; config desabilitada→noop; max_turns→noop.

**Exit:** `decideSdrAction` coberto por testes de todos os ramos; **sem** side-effects (nada de send/queue/db-write); tsc/eslint limpos.

## SDR-6 — Worker (consumer) + envio/persistência + guardas

**Contexto cold-start.** O consumidor da fila que executa a decisão do SDR-5. Roda como processo separado, **service-role** (sem sessão de usuário).

**Tarefas.**
1. `src/worker/sdr-worker.ts`: BullMQ `Worker` na fila `sdr` (conexão do SDR-3). Por job:
   - `decision = decideSdrAction(deps, { conversationId })`.
   - `reply`: resolve o canal da conversa (`channels` row via `channel_id`) → `createChannel(row).sendText({ to, text })` → persiste `messages` (sender_type 'bot') → bump `conversations` → grava `sdr_runs`.
   - `handoff`: `sdr_status='handoff'`, **notifica** (ver mecanismo abaixo), grava run, **não** responde.
   - `noop`: grava run e encerra.
2. **Guardas server-side:** janela 24h no Cloud (rejeitar freeform fora da janela — usar `channelCapabilities`/lógica de janela; Evolution é freeform livre); rate limit por conversa; try/catch por job com `sdr_runs.action='error'` + `attempts`/backoff do BullMQ.
3. Entry script (default **`tsx`** — roda TS direto, sem bundle separado, e compartilha o código do app): add dep `tsx`; `package.json` → `"worker": "tsx src/worker/sdr-worker.ts"`. (Alternativa só se `tsx` pesar: bundle esbuild. Não usar Next standalone pro worker — ele não empacota entry custom.)
4. **Mecanismo de notificação (handoff/erro):** config por account de um "número de notificação" (campo no account/settings, E.164) + canal a usar. O worker envia via `createChannel(canalDeNotificacao).sendText(numeroNotificacao, msg)`. Se não configurado → só grava o run + log (não falha o job). Definir o campo de config aqui (migração pequena ou reaproveitar settings existentes).
5. Teste de integração do handler (mock canal/queue/db): reply envia+persiste+loga; handoff muda status+notifica+não envia; erro vira run 'error'.

**Exit:** worker processa os 3 ramos com guardas; logs em `sdr_runs`; testes do handler; tsc/eslint/build limpos. (Deploy é o SDR-7.)

## Riscos & mitigações (catálogo)
- **R1 — Conexão Redis em import-time** quebra o build Next. → conexão lazy (SDR-3).
- **R2 — ENCRYPTION_KEY divergente** entre web e worker → worker não decifra `channels.credentials` → envio falha. → mesma chave, documentado (SDR-7).
- **R3 — Loop do bot** (responder a si mesmo / a outro bot) → guarda de `fromMe`/sender + `max_turns` + Filtro Anti-Bot (SDR-5/6). *(o n8n tinha "Filtro Anti-Bot" — preservar a ideia.)*
- **R4 — Janela 24h Cloud** → guarda server-side antes de chamar a Meta (SDR-6).
- **R5 — Duplicidade de processamento** (duas réplicas do worker) → BullMQ garante 1 consumer por job; `jobId` por conversa evita corrida de debounce.
- **R6 — Config por campanha ausente** quando a conversa não veio de campanha → SDR fica `off` por default; sem campanha, sem SDR (decisão explícita).
- **R7 — Custo LLM/transcrição** → `max_turns`, debounce, e log de tokens em `sdr_runs` pra auditar.
